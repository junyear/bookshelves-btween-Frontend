“책은 읽었는데, 이 감상을 누구랑 나누지?”
“독서모임은 부담스럽고, 혼자 기록하기엔 아쉬워요.”

책을 읽고 나면 누군가와 이야기하고 싶은 순간이 있습니다.

하지만 기존 독서모임은 정기적으로 참여해야 하거나, 새로운 사람들과 관계를 만들어야 한다는 부담이 있습니다.

책을 좋아하지만 독서모임이 부담스러웠던 사람들을 위해, **익명으로 가볍게 참여하고 책 이야기만 남길 수 있는 서비스**를 만들고자 합니다.

## 🔖 브랜치 컨벤션
<!-- 팀원들끼리 협의하여 기록해주세요! -->

본 프로젝트는 Fork 기반 협업 방식을 사용합니다.  

각 팀원은 Upstream Repository를 Fork한 개인 Origin Repository에서 작업한 뒤, Pull Request를 통해 Upstream Repository에 반영합니다.

### Repository 구조

- `upstream`: 팀 공용 원본 저장소

- `origin`: 각 팀원의 개인 Fork 저장소

- `local`: 각 팀원의 로컬 개발 환경

### 작업 흐름

1. Upstream Repository를 Fork하여 개인 Origin Repository를 생성합니다.

2. 개인 Origin Repository를 로컬 환경에 Clone합니다.

3. 작업 전 Upstream Repository의 최신 변경 사항을 Pull 받아 동기화합니다.

4. 기능별 브랜치를 생성하여 작업합니다.

5. 작업 완료 후 개인 Origin Repository로 Push합니다.

6. Upstream Repository로 Pull Request를 생성합니다.

7. 코드 리뷰 후 문제가 없으면 Upstream Repository에 Merge합니다.

## 🌀 코딩 컨벤션

본 프로젝트의 iOS 파트는 StyleShare Swift Style Guide를 기준으로 코드를 작성합니다.
문서에 명시되지 않은 세부 규칙은 Swift API Design Guidelines를 따릅니다.

### 기본 규칙

- 들여쓰기는 Tab 대신 Space 2칸을 사용합니다.
- 한 줄은 최대 99자를 넘지 않도록 작성합니다.
- 콜론(:)은 오른쪽에만 공백을 둡니다.
  - Good: let name: String
  - Bad: let name : String
- 빈 줄에는 공백이 포함되지 않도록 합니다.
- 모든 파일은 빈 줄로 끝나도록 합니다.
- 불필요한 주석은 작성하지 않고, 코드만으로 의도가 드러나도록 작성합니다.

### Import 규칙

- Import는 알파벳 순으로 정렬합니다.
- 기본 프레임워크를 먼저 작성합니다.
- 외부 라이브러리가 있는 경우 기본 프레임워크와 한 줄 띄워 구분합니다.

예시:

import Foundation
import SwiftUI
import UIKit

import Alamofire
import Kingfisher

### Naming 규칙

- Class, Struct, Enum, Protocol 이름은 UpperCamelCase를 사용합니다.
  - Good: HomeView, ItemModel, ItemCategory, ItemRepository
  - Bad: homeView, item_model, itemcategory

- Function, Variable, Constant 이름은 lowerCamelCase를 사용합니다.
  - Good: fetchItems(), itemCount, selectedItemName
  - Bad: FetchItems(), item_count, SelectedItemName

- Bool 값은 의미가 명확하게 드러나도록 작성합니다.
  - Good: isSelected, hasImage, canSubmit
  - Bad: selected, image, submit

### Function 규칙

- 함수 이름 앞에는 불필요하게 get을 붙이지 않습니다.
  - Good: func itemName(for item: Item) -> String
  - Bad: func getItemName(for item: Item) -> String

- Action 함수는 주어 + 동사 + 목적어 형태로 작성합니다.
  - Good: registerButtonDidTap()
  - Good: itemCellDidTap()
  - Bad: register()
  - Bad: tapItem()

- Tap은 눌렀다가 뗀 동작에 사용합니다.
- Press는 누르는 동작에 사용합니다.
- will은 특정 동작이 일어나기 직전에 사용합니다.
- did는 특정 동작이 일어난 직후에 사용합니다.
- should는 Bool을 반환하는 함수에 사용합니다.

### Enum 규칙

- Enum 이름은 UpperCamelCase를 사용합니다.
- Enum case 이름은 lowerCamelCase를 사용합니다.

예시:

enum ItemStatus {
  case available
  case reserved
  case completed
}

### Closure 규칙

- 파라미터와 리턴 타입이 없는 클로저는 () -> Void를 사용합니다.
  - Good: let completion: (() -> Void)?
  - Bad: let completion: (() -> ())?

- 클로저에서 타입 추론이 가능한 경우 타입 선언을 생략합니다.
- 클로저 정의 시 파라미터에는 불필요한 괄호를 사용하지 않습니다.

예시:

UIView.animate(withDuration: 0.3) {
  self.view.alpha = 1.0
}

### Type 규칙

- Array와 Dictionary는 축약형을 사용합니다.
  - Good: var items: [Item]
  - Good: var itemMap: [Int: Item]
  - Bad: var items: Array<Item>
  - Bad: var itemMap: Dictionary<Int, Item>

### Self 규칙

- 클래스와 구조체 내부에서는 self를 명시적으로 사용합니다.

예시:

final class ItemViewModel {
  private var items: [Item] = []

  func updateItems(_ items: [Item]) {
    self.items = items
  }
}

### Class 규칙

- 상속이 필요하지 않은 클래스는 final 키워드를 사용합니다.

예시:

final class HomeViewModel {
}

### MARK 규칙

- 관련된 코드는 // MARK: 를 사용하여 구분합니다.
- MARK 구문 위와 아래에는 빈 줄을 둡니다.

예시:

// MARK: - Properties

private let titleLabel = UILabel()

// MARK: - Init

override init(frame: CGRect) {
  super.init(frame: frame)
}

// MARK: - Actions

func submitButtonDidTap() {
}

### Protocol 규칙

- 프로토콜 이름은 UpperCamelCase를 사용합니다.
- 프로토콜을 적용할 때는 extension을 사용하여 관련 메서드를 분리합니다.

예시:

final class HomeViewController: UIViewController {
}

// MARK: - UITableViewDataSource

extension HomeViewController: UITableViewDataSource {
}

// MARK: - UITableViewDelegate

extension HomeViewController: UITableViewDelegate {
}

### Delegate 규칙

- Delegate 메서드는 프로토콜명으로 네임스페이스를 구분합니다.
- 의미가 모호한 이름은 사용하지 않습니다.

예시:

protocol UserCellDelegate {
  func userCellDidSetProfileImage(_ cell: UserCell)
  func userCell(_ cell: UserCell, didTapFollowButtonWith user: User)
}

### Constant 규칙

- 비슷한 상수는 enum으로 묶어 관리합니다.
- 상수 이름은 lowerCamelCase를 사용합니다.
- MAX_COUNT와 같은 대문자 스네이크 케이스는 사용하지 않습니다.

예시:

private enum Metric {
  static let horizontalPadding: CGFloat = 16
  static let cornerRadius: CGFloat = 12
}

private enum Font {
  static let title = UIFont.boldSystemFont(ofSize: 18)
  static let body = UIFont.systemFont(ofSize: 14)
}

### Comment 규칙

- 문서화가 필요한 주석은 ///를 사용합니다.
- 단순 설명용 주석은 최소화합니다.
- 코드의 의도는 네이밍과 구조로 드러나도록 작성합니다.

예시:

/// 물품 목록을 보여주는 화면
final class ItemListViewController: UIViewController {
}

### SwiftUI 규칙

- SwiftUI View 이름은 화면 또는 컴포넌트의 역할이 드러나도록 작성합니다.
  - Good: HomeView, ItemListView, ItemDetailView, ItemCardView
  - Bad: MainView, CustomView, TestView

- ViewModel은 화면 이름 뒤에 ViewModel을 붙입니다.
  - Good: HomeViewModel, ItemListViewModel
  - Bad: HomeVM, ItemManager

- 재사용 가능한 View 컴포넌트는 역할이 드러나도록 작성합니다.
  - Good: ItemCardView, PrimaryButton, EmptyStateView
  - Bad: CommonView, CustomButton, BoxView

### File Naming 규칙

- 파일 이름은 내부에 선언된 주요 타입 이름과 동일하게 작성합니다.
  - HomeView.swift
  - HomeViewModel.swift
  - ItemModel.swift
  - ItemCardView.swift

### Commit 전 확인 사항

- 들여쓰기는 Space 2칸으로 맞췄는가
- 한 줄이 99자를 넘지 않는가
- 사용하지 않는 코드가 남아있지 않은가
- 불필요한 주석이 남아있지 않은가
- 함수, 변수, 타입 이름이 명확한가
- Import가 정리되어 있는가
- 관련 코드를 MARK로 구분했는가
- 상속이 필요 없는 클래스에 final을 붙였는가
- SwiftUI View와 ViewModel 파일명이 역할에 맞게 작성되었는가

## 📁 PR 컨벤션

본 프로젝트는 Pull Request를 통해 코드 리뷰 후 `main` 브랜치에 병합합니다.  
모든 작업은 개인 브랜치에서 진행한 뒤 Pull Request를 생성합니다.

### PR 제목 규칙

PR 제목은 아래 형식을 따릅니다.

[Type] 작업 내용 요약

#### 예시

[feat] 물품 등록 화면 구현  
[fix] 로그인 버튼 동작 오류 수정  
[design] 홈 화면 UI 수정  
[refactor] ItemViewModel 구조 개선  
[docs] README 컨벤션 문서 수정  
[hotfix] 배포 후 치명적 오류 긴급 수정  

### PR Type

| Type | 설명 |
|---|---|
| feat | 새로운 기능 추가 |
| fix | 버그 수정 |
| hotfix | 배포 후 발생한 긴급 오류 수정 |
| design | UI 디자인 수정 |
| refactor | 코드 리팩토링 |
| chore | 프로젝트 설정, 빌드 설정, 파일 정리 등 |
| docs | 문서 수정 |
| test | 테스트 코드 추가 또는 수정 |

### PR 작성 규칙

- PR은 하나의 기능 또는 하나의 작업 단위로 생성합니다.
- PR 제목만 보고 어떤 작업인지 알 수 있도록 작성합니다.
- 작업한 내용을 간단하고 명확하게 작성합니다.
- UI 변경이 있는 경우 스크린샷을 첨부합니다.
- 확인이 필요한 부분이 있다면 PR 본문에 따로 작성합니다.
- 관련 이슈가 있다면 이슈 번호를 함께 작성합니다.
- 충돌이 발생한 경우 해결 후 리뷰를 요청합니다.

## 📑 커밋 컨벤션

본 프로젝트는 커밋 메시지에 별도의 타입을 강제하지 않고, 작업 내용을 자유롭게 작성합니다.  
다만 커밋 기록만 보고 어떤 작업이 이루어졌는지 이해할 수 있도록 명확하게 작성합니다.

### Commit 작성 규칙

- 커밋 메시지는 한글로 작성합니다.
- 커밋 메시지는 작업 내용을 짧고 명확하게 작성합니다.
- 하나의 커밋에는 하나의 작업 단위만 포함합니다.
- 너무 포괄적인 표현은 사용하지 않습니다.
- 커밋 제목 끝에는 마침표를 붙이지 않습니다.
- 브랜치 목적과 관련 없는 파일은 커밋하지 않습니다.

### Commit Message Example

#### 좋은 예시

```text
물품 목록 화면 구현
물품 상세 화면 UI 수정
등록 버튼 색상 변경
카테고리 선택 오류 수정
홈 화면 ViewModel 로직 분리
README 컨벤션 문서 작성
사용하지 않는 이미지 파일 삭제
프로젝트 폴더 구조 정리
```
